# Template T12 — Expertise card

## Rôle

Slide qui présente **une expertise Frontguys** avec sa promesse, ses domaines de compétence et sa valeur ajoutée. Format reproductible — un slide par expertise dans une section "Nos expertises".

## Structure visuelle

```
┌─────────────────────────────────────────────┐
│   <Nom de l'expertise>            (H1)      │
│   <Promesse / sous-titre>          (H3)     │
│                                             │
│   Expertises :                              │
│   • <Compétence 1>                          │
│   • <Compétence 2>                          │
│   • <Compétence 3>                          │
│   • <Compétence 4>                          │
│                                             │
│   Notre valeur ajoutée :                    │
│   ▶ <Bénéfice 1>                            │
│   ▶ <Bénéfice 2>                            │
│   ▶ <Bénéfice 3>                            │
└─────────────────────────────────────────────┘
```

## Contenu type — Modernisation et migration front-end

**Titre H1** : Modernisation des applications et migration front-end

**Sous-titre H3** : Réduire la dette technique et unifier les technologies front-end pour des produits plus performants, évolutifs et cohérents.

**Expertises** :

* Migration de frameworks (Backbone, AngularJS vers Angular, Vue ou Next.js)
* Structuration et intégration de Design Systems robustes (Storybook, Figma, tokens)
* Refonte complète de parcours utilisateurs
* Mise en place de tests automatisés (unitaires, visuels, end-to-end)
* Amélioration continue des performances et du SEO

**Notre valeur ajoutée** :

* ▶ Accélération de la transformation digitale
* ▶ Expérience utilisateur améliorée : refonte des parcours critiques fondée sur la data, les usages réels et les bonnes pratiques UX
* ▶ Durabilité du produit : architectures front-end modernes et maintenables

## Contenu type — Design System (RAO BPI)

**Titre H1** : Design System, collaboration Design/Dev et au-delà

**Sous-titre H3** : Créer une cohérence visuelle et technique sur l'ensemble des produits numériques.

**Expertises** :

* Mise en place, documentation et maintenance de Design Systems
* Intégration et automatisation des Design Tokens
* Accompagnement des équipes dans l'adoption des composants réutilisables
* Processus de revue Design–Dev et audits de cohérence UI/UX

**Notre valeur ajoutée** :

* ▶ Cohérence et efficacité à l'échelle : un langage visuel et technique partagé
* ▶ Réduction du time-to-market : industrialisation des processus
* ▶ Meilleure collaboration interdisciplinaire : alignement durable entre design, produit et dev

## Variables / paramètres

* **Titre H1** : nom de l'expertise (court, descriptif)
* **Sous-titre H3** : promesse / résultat attendu (1 phrase, "pour \[bénéfice\]")
* **Section "Expertises"** : 4-6 compétences listées (formulation nominale + verbale concrète)
* **Section "Notre valeur ajoutée"** : 3 bénéfices clients structurés, formulés en termes business

## Règles de voix

* **Titre** : nominal long autorisé (l'expertise a souvent un nom composé)
* **Sous-titre** : infinitif d'action ("Réduire", "Créer", "Accélérer") + bénéfice — pattern récurrent Frontguys
* **Expertises (liste 1)** :
  * Formulation **nominale** (noms d'action : "Migration", "Structuration", "Refonte")
  * Précisions techniques entre parenthèses si pertinent
  * Parallélisme syntaxique sur les 4-6 items
* **Valeur ajoutée (liste 2)** :
  * Format "▶ <Concept clé> : <explication concrète>"
  * Vocabulaire business (time-to-market, conformité, accélération, ROI)
  * 3 bénéfices maximum (au-delà, l'argument se dilue)

## Anti-patterns

* ❌ Trop d'expertises (> 6) — l'expertise paraît trop large, perdue
* ❌ "Notre valeur ajoutée" avec plus de 3 items — l'argument se dilue
* ❌ Bénéfices vagues ("amélioration de la qualité") — préférer du chiffrable
* ❌ Auto-valorisation gratuite ("Notre excellence en…") — préférer une description factuelle
* ❌ Mélange Expertises / Valeur ajoutée dans un seul bloc (la séparation est structurante)
* ❌ Expertise sans valeur ajoutée — toujours expliquer "ce que ça apporte au client"

## Pattern Frontguys récurrent — Les 4 expertises canoniques

Observées dans le RAO BPI :

| Expertise | Promesse |
|-----------|----------|
| **Modernisation et migration front-end** | Réduire la dette technique et unifier les technologies |
| **Design System, collaboration Design/Dev** | Créer une cohérence visuelle et technique |
| **Accessibilité numérique et conformité RGAA/EAA** | Garantir la conformité légale |
| **Performance d'équipe et pilotage technique** | Accélérer la livraison et la qualité |

Chacune suit le même pattern de slide T12 — modèle reproductible.

## Variantes

### Variante "Expertise + cas client"

Sous le bloc "Notre valeur ajoutée", ajouter une mini-section "Exemple :" qui pointe vers un cas client précis.

### Variante "Expertise + chiffres"

Ajouter un sous-bloc "Chiffres clés" (1-3 chiffres marquants) au-dessus de "Notre valeur ajoutée".

### Variante "Expertise + équipe"

Mentionner les profils consultants typiques mobilisés pour cette expertise.
